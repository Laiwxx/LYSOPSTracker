## SECTION 2 — BACKEND ROUTES AUDIT

---

FILE: server.js (all cron job handlers — lines 1732, 1869, 1893, 1924, 1948)  
ISSUE: None of the cron job functions have a top-level try/catch. If `readTasks()`, `readProjects()`, `readEOD()`, or `readClaims()` throws (e.g., due to a corrupted JSON file), the async cron callback will throw an unhandled exception. `node-cron` does not crash the process for this, but the cron silently fails and no error is logged.  
SEVERITY: Medium  
FIX: Wrap each cron callback body in `try { ... } catch (e) { logError('cron.name', e); }`.

---

FILE: server.js — `PUT /api/projects/:id` (line 650), `PUT /api/projects/:id/fabrication/:idx` (line 799), `PUT /api/projects/:id/installation/:idx` (line 838), `PUT /api/projects/:id/delivery-requests/:reqId` (line 858), `PUT /api/claims/:id` (line 1641)  
ISSUE: These PUT routes accept `req.body` directly via spread/Object.assign with no field allowlisting. Any arbitrary key can be injected into the stored JSON objects. For example, a crafted request could inject `__proto__` or any fake property into a project record.  
SEVERITY: Medium  
FIX: Use an explicit field allowlist for each PUT route rather than spreading `req.body` directly.

---

FILE: server.js — `DELETE /api/projects/:id` (line 717)  
ISSUE: The server-side DELETE project route has no PIN verification. The PIN check is performed only in the client-side JavaScript (`project.js` line 234–247). Any user with network access to the server can delete any project with a simple `DELETE /api/projects/{id}` request — bypassing the browser entirely.  
SEVERITY: High  
FIX: Add server-side PIN verification to this route. Read the PIN from `req.headers['x-admin-pin']` or `req.body.pin`, verify against `readAdmin().pin`, and reject with 403 if it doesn't match. Update the client to include the PIN in the delete request header/body.

---

FILE: server.js — `DELETE /api/staff/:name` (line 437)  
ISSUE: No server-side auth. Any caller can remove a staff member.  
SEVERITY: High  
FIX: Add PIN verification as above.

---

FILE: server.js — `DELETE /api/claims/:id` (line 1662), `DELETE /api/tasks/:id` (line 1472), `DELETE /api/workers/:id` (line 2130), `DELETE /api/tickets/:id` (line 404), `DELETE /api/suppliers/:id` (line 2759)  
ISSUE: No server-side auth on any of these delete routes.  
SEVERITY: Medium  
FIX: Add PIN verification for at minimum claims (financial data) and staff. Tasks/tickets can be lower priority.

---

FILE: server.js — `POST /api/admin/recalc` (line 640)  
ISSUE: This route recalculates and overwrites derived fields for ALL projects. No PIN protection. Could be triggered by any user to force recalculation — low risk but inconsistent with admin-only intent.  
SEVERITY: Low  
FIX: Add PIN header check.

---

FILE: server.js — Missing route: `DELETE /api/purchase-orders/:id`  
ISSUE: There is no DELETE route for purchase orders. The Procurement module can create and update POs but cannot delete them. This means erroneous or duplicate POs accumulate permanently.  
SEVERITY: High  
FIX: Add `app.delete('/api/purchase-orders/:id', ...)` following the same pattern as `DELETE /api/suppliers/:id`.

---

FILE: server.js — `/api/remind` route (line 1065)  
ISSUE: This route calls the Microsoft Graph API directly rather than using `sendEmail()`. This means it bypasses `EMAIL_TEST_OVERRIDE`. During testing, clicking the "!" Remind button in the NEEDS ATTENTION panel sends real emails to the actual stage owner rather than routing to the test override address. This is inconsistent with all other email sends.  
SEVERITY: Medium  
FIX: Refactor `/api/remind` to call `sendEmail()` instead of raw Graph API fetch. This will automatically apply the test override and error logging.

---

FILE: server.js — `GET /api/eod-log` (line 1680)  
ISSUE: `getStaffNames()` is called on every request to this endpoint, which reads and parses `staff.json` every time. Additionally, `readTasks()` is called inside cron jobs that also call `readProjects()` and `readEOD()` in the same tick — all separate file reads.  
SEVERITY: Low  
FIX: Acceptable for current scale. If the app grows, add a simple in-memory cache with a 60s TTL for staff data.
