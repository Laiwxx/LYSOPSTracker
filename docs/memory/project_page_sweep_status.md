- **Phase C finalize — the dual-path bug that almost killed Phase C silently:** The full-project PUT handler (`PUT /api/projects/:id`) does a shallow spread `{ ...oldProject, ...incoming }`. When `/project.html`'s `buildFabRow` sync writes fab fields and calls `debouncedSave()`, the payload does NOT include `logs[]`, so the spread would REPLACE the server-side fabrication array and wipe every log entry on every save. This became load-bearing the instant Phase C shipped because live logs were being created and any parallel /project.html edit would silently destroy them. **Fix:** the PUT handler now walks `incoming.fabrication[]` before the spread; for each row, if fields are absent (`logs`, `fab_started_at`, `fab_completed_at`, `cycle_days`), it copies them from the matching old row (matched by item-name first, index fallback). Verified via smoke test: client sends project with `logs` explicitly deleted → server preserves logs[] on disk. Closes the known dual-path gap for the log field specifically; the `qtyDone` write path from /project still exists as documented technical debt until Phase 4 retires the /project fab editor.
